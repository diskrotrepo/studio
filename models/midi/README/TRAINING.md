# Training

## Pre-tokenization

Pre-tokenization validates MIDI files and caches tokens. It's:
- **Required** for multi-GPU training (avoids distributed timeout issues)
- **Recommended** for single-GPU (catches corrupted files that would crash training)

```bash
# Validate and pre-tokenize
python3 pretokenize.py --validate-only
./scripts/delete_bad_midi.sh  # if bad files found
python3 pretokenize.py
```

### Pre-tokenization Options

| Argument | Default | Description |
|----------|---------|-------------|
| `--midi-dir` | `midi_files` | Directory containing MIDI files |
| `--output` | `checkpoints/token_cache.pkl` | Output cache file path |
| `--no-tags` | False | Disable tag conditioning |
| `--workers` | Auto | Number of worker processes (CPU count / 2) |
| `--parallel` | False | Force parallel processing on macOS (may crash) |
| `--skip-validation` | False | Skip MIDI file validation (not recommended) |
| `--validate-only` | False | Only validate files, don't tokenize |
| `--single-track` | False | Use single-track tokenization instead of multi-track |
| `--max-tracks` | 16 | Maximum tracks per file (for multi-track mode) |
| `--checkpoint-interval` | 500 | Save progress every N files |
| `--metadata` | None | Path to JSON metadata file (e.g., midi_flat/midi_metadata.json) |
| `--max-files` | None | Limit number of files to process (for smoke-testing) |

Examples:
```bash
python3 pretokenize.py --midi-dir /path/to/midi    # Custom MIDI directory
python3 pretokenize.py --no-tags                    # Disable tag conditioning
python3 pretokenize.py --validate-only              # Just find bad files
python3 pretokenize.py --workers 4                  # Limit parallel workers
python3 pretokenize.py --single-track                # Single-track tokenization
python3 pretokenize.py --max-tracks 8                # Limit to 8 tracks per file
python3 pretokenize.py --max-files 50                # Smoke-test with 50 files
```

### MIDI File Validation

Pre-tokenization automatically validates MIDI files before processing. This catches corrupted files that would crash the symusic library (segmentation faults in C++ code can't be caught by Python).

Generated files:
- `midi_processing.log` - Real-time log of each file being processed (check this to find crash-causing files)
- `bad_midi_files.txt` - Log of all corrupted files with error messages
- `scripts/delete_bad_midi.sh` - Watches logs and deletes bad MIDI files in real-time

If processing crashes, check the last `VALIDATING:` or `TOKENIZING:` line in `midi_processing.log` without a corresponding `SUCCESS:` - that's the file that caused the crash.

Workflow for cleaning a dataset:
```bash
# Find all corrupted files
python3 pretokenize.py --validate-only

# Review the bad files (optional)
cat bad_midi_files.txt

# Remove them
./scripts/delete_bad_midi.sh

# Now tokenize the clean dataset
python3 pretokenize.py
```

## Organize by Genre (Optional)

If your MIDI files are organized by artist folders, you can automatically reorganize them by genre using the MusicBrainz API:

```bash
pip install musicbrainzngs
python organize_by_genre.py
```

This creates a `midi_by_genre/` folder with subfolders for each genre:
- Rock, Pop, Electronic, Jazz, Classical, Metal, Hip-Hop, R&B-Soul
- Country, Blues, Reggae, Latin, World, Soundtrack, Easy Listening, Unknown

The script is resumable - it caches artist lookups in `genre_cache.json`, so you can restart if interrupted.

## Sample by Genre

If you have a `midi_metadata.json` file (generated by the Last.fm tag pipeline), you can create training subsets by genre using `scripts/sample_by_genre.py`. This is useful for training genre-specific models without manually sorting files.

```bash
# List all available genres and file counts
python3 scripts/sample_by_genre.py --list-genres

# Sample 3000 random electronic files
python3 scripts/sample_by_genre.py --genre electronic --count 3000

# Sample 1000 files from each of multiple genres
python3 scripts/sample_by_genre.py --genre electronic --genre jazz --count 1000

# Sample 500 from every genre
python3 scripts/sample_by_genre.py --all-genres --count 500
```

The tool creates a `midi_subset/` directory (configurable with `--output-dir`) containing symlinks to the original MIDI files and a filtered `midi_metadata.json` with genre/mood tags preserved. Then tokenize and train on the subset as usual:

```bash
python3 pretokenize.py --midi-dir midi_subset --metadata midi_subset/midi_metadata.json
python3 train.py --midi-dir midi_subset
```

| Argument | Default | Description |
|----------|---------|-------------|
| `--list-genres` | - | List all genres and file counts, then exit |
| `--genre` | - | Genre to sample (repeatable, e.g. `--genre rock --genre pop`) |
| `--all-genres` | False | Sample from every genre |
| `--count` | 1000 | Number of files to sample per genre |
| `--output-dir` | `midi_subset` | Output directory for the subset |
| `--metadata` | `midi_files/midi_metadata.json` | Path to metadata JSON |
| `--midi-dir` | `midi_files` | Source directory containing MIDI files |
| `--seed` | 42 | Random seed for reproducibility |
| `--copy` | False | Copy files instead of creating symlinks |

Genre names are flexible — `electronic`, `Electronic`, and `GENRE_ELECTRONIC` all work. Files belonging to multiple genres are deduplicated in the output.

## Single-GPU Training

```bash
# Train with a hardware config preset
python3 train.py --config configs/a100_40gb.json
python3 train.py --config configs/m4_max_128gb.json

# Train on a specific genre
python3 pretokenize.py --midi-dir midi_by_genre/Rock
python3 train.py --config configs/a100_40gb.json --midi-dir midi_by_genre/Rock --checkpoint-dir checkpoints/rock

# Override specific config values via CLI
python3 train.py --config configs/a100_40gb.json --batch-size 8 --lr 1e-4
```

This will:
- Tokenize all MIDI files using REMI encoding
- Train a 12-layer transformer model
- Save checkpoints to `checkpoints/`

### Training Options

| Argument | Default | Description |
|----------|---------|-------------|
| `--config` | None | Path to JSON config file (e.g., `configs/a100_40gb.json`) |
| `--midi-dir` | `midi_files` | Directory containing MIDI files |
| `--checkpoint-dir` | `checkpoints` | Directory for saving model checkpoints |
| `--load-from` | None | Path to pre-trained checkpoint to fine-tune from |
| `--finetune` | False | Fine-tune mode: reset epoch counter and use fresh optimizer |
| `--lr` | from config | Learning rate (overrides config; default 3e-4, fine-tune 3e-5) |
| `--freeze-layers` | 0 | Number of transformer layers to freeze from the bottom |
| `--lora` | False | Enable LoRA training (freezes base model, trains small adapter) |
| `--lora-rank` | 8 | LoRA rank - higher = more capacity but more params |
| `--lora-alpha` | 16.0 | LoRA alpha scaling factor |
| `--batch-size` | from config | Per-GPU batch size (overrides config) |
| `--grad-accum` | from config | Gradient accumulation steps (overrides config) |
| `--max-files` | None | Limit number of sequences to train on (for smoke-testing) |

## Multi-GPU Training

After pre-tokenization, run distributed training:

```bash
torchrun --nproc_per_node=8 train.py --config configs/8x_a100.json
```

More examples:
```bash
# 4 GPUs
torchrun --nproc_per_node=4 train.py --config configs/8x_a100.json

# Multi-node (example: 2 nodes with 8 GPUs each)
torchrun --nnodes=2 --node_rank=0 --master_addr=<master_ip> --master_port=29500 --nproc_per_node=8 train.py --config configs/8x_a100.json
```

The script automatically:
- Shards data across GPUs using `DistributedSampler`
- Wraps the model with `DistributedDataParallel`
- Handles logging/checkpointing on rank 0 only
- Scales learning rate for larger effective batch sizes

## Multi-Track Training

For training with explicit multi-track support (up to 16 tracks, ~6 minute songs):

```bash
# 1. Pre-tokenize (multi-track is the default)
python3 pretokenize.py

# 2. Train (uses MultiTrackMusicTransformer automatically when cache has multi-track data)
python3 train.py
```

The multi-track model uses:
- **Track embeddings**: Each token knows which instrument/track it belongs to (16 tracks max)
- **Cross-track attention**: Tokens attend to other tracks at the same bar position
- **Track-type conditioning**: 15 roles (melody, bass, chords, drums, percussion, pad, strings, lead, organ, brass, woodwind, synth, choir, fx, other)
- **Long context**: 8192 tokens supports ~6 minute songs with full orchestration

### Rich Conditioning Tags

The model extracts 75 musical attribute tags from MIDI files automatically during pre-tokenization across 15 categories:

| Category | Auto-Detected From |
|----------|-------------------|
| Key signature | MIDI key signature events |
| Time signature | MIDI time signature events |
| Note density | Notes per beat ratio |
| Dynamics | Velocity distribution |
| Pitch register | Average pitch range |
| Arrangement size | Number of active tracks |
| Rhythmic feel | Note onset distribution |
| Harmonic complexity | Unique pitch classes |
| Articulation | Note duration vs gaps |
| Expression | Velocity variance + pitch bends |
| Era/Style | Instrument program mix |

This enables fine-grained control during generation:
```bash
python3 generate.py --tags "jazz minor 3_4 swing expressive vintage small_ensemble"
```

## Configuration

Training is configured via JSON preset files in `configs/`. Use `--config` to select one:

```bash
python3 train.py --config configs/a100_40gb.json
python3 train.py --config configs/m4_max_128gb.json
torchrun --nproc_per_node=8 train.py --config configs/8x_a100.json
```

CLI arguments (`--batch-size`, `--grad-accum`, `--lr`) override config file values.

### Hardware Presets

All presets use the same model architecture (512 d_model, 8 heads, 12 layers) for checkpoint compatibility. They differ in batch size, gradient accumulation, and sequence length.

| Setting | M4 Max 128GB | A100 40GB | 8x A100 |
|---------|-------------|-----------|---------|
| seq_length | 2048 | 8192 | 8192 |
| batch_size_per_gpu | 16 | 12 | 6 |
| gradient_accumulation | 8 | 5 | 1 |
| learning_rate | 3e-4 | 1e-4 | 1e-4 |
| dropout | 0.1 | 0.2 | 0.2 |
| use_compile | false | true | true |
| distributed_timeout | 30 min | 30 min | 60 min |
| **effective batch** | **128** | **60** | **48** |

### Custom Configs

Copy an existing preset and modify it:

```bash
cp configs/a100_40gb.json configs/my_custom.json
# Edit the new file, then:
python3 train.py --config configs/my_custom.json
```

Config files are JSON with all `TrainingConfig` fields. Keys prefixed with `_` (like `_description`) are ignored and can be used for comments.

## Full Pipeline on Single A100 GPU

Complete workflow from setup to generation on a single A100:

```bash
# 1. Environment setup
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 2. Prepare your MIDI dataset
mkdir midi_files
# Add your .mid files to midi_files/

# 3. Validate and clean the dataset
python3 pretokenize.py --validate-only
./scripts/delete_bad_midi.sh  # Remove any corrupted files

# 4. Smoke-test the pipeline with a small subset first
python3 pretokenize.py --max-files 50
python3 train.py --config configs/a100_40gb.json --max-files 50

# 5. Pre-tokenize and train the full dataset
python3 pretokenize.py
python3 train.py --config configs/a100_40gb.json

# 6. Generate music
python3 generate.py --tags "jazz energetic"
```

**Expected training time on A100 (40GB):**
- ~1,000 MIDI files: 2-4 hours for 20 epochs
- ~10,000 MIDI files: 8-12 hours for 20 epochs
- ~100,000 MIDI files: 2-3 days for 20 epochs

Monitor GPU usage with `nvidia-smi` - the A100 should show high utilization (>90%). If memory usage is low, increase batch size with `--batch-size`.

### Running on a Remote Machine

To keep training running after closing your SSH session, use `tmux`:

```bash
# Start a new tmux session
tmux new -s train

# Activate environment and start training
python3 -m venv venv
source venv/bin/activate
python3 train.py

# Detach from session: press Ctrl+B, then D
# You can now safely close SSH

# Reconnect later to check progress:
tmux attach -t train
```

If tmux isn't installed: `sudo apt install tmux`

## Tips for Better Results

1. **More data is better** - Try to get at least a few hundred MIDI files
2. **Filter your dataset** - Similar styles train faster (e.g., just piano, just classical)
3. **Longer training** - Music models benefit from many epochs
4. **Larger models** - If you have a GPU, try 512 dimensions and 8+ layers
5. **Temperature tuning** - Start at 1.0, adjust based on output quality
6. **Use tags effectively** - Tags work best when your training data includes diverse styles; the model learns to associate tags with musical patterns
7. **Prompt continuation** - For extending a piece, the model uses the first 256 tokens as context; longer prompts give more stylistic continuity
